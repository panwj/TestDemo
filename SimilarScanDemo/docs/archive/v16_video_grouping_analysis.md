## 2. 23 个大组的原因

每个普通视频通常有 13 个等距采样帧。`OPTION_CLOSEST_SYNC` 会把多个相邻采样时间点
映射到同一个视频关键帧，因此一段视频的帧列表中可能出现多个相同或近似画面。

旧比较方式：

```text
左侧每一帧 x 右侧每一帧
任意命中两次即相似
```

问题是同一张左侧画面可以同时命中右侧两个重复关键帧：

```text
左帧 A -> 右帧 B1
左帧 A -> 右帧 B2
```

即使两段视频只有一处公共画面，也会满足“两次命中”，导致清理应用界面、相同背景、
相同片头或静止画面的录屏被合并进一个超大组。

## 4. v16 改造

### 4.1 对应位置比较

视频采样点本身是按视频时长归一化生成的：

```text
0%, 8.3%, 16.7%, ... 100%
```

因此比较：

```text
framesA[index] 与 framesB[index]
```

可以避免一张重复关键帧多次贡献命中，同时保留不同视频时长之间的相对时间对齐。

仍然使用：

```text
至少两个位置命中
```

单帧 dHash、colorHash 和阈值没有修改。

### 4.2 视频锚点顺序

竞品视频专用流程保留 MediaStore `date_added DESC` 输入顺序。Demo 当前数据库没有
单独保存 `date_added`，v16 暂时使用：

```text
media_store_id DESC
```

作为最接近的稳定顺序。

图片和截图仍使用：

```text
created_at ASC, media_store_id DESC
```

不会受本次视频分组调整影响。

### 4.3 数据库升级

数据库版本升级至 v16，旧视频分组和指纹会全量重建。

## 5. 仍需验证

安装 v16 后重新全量扫描，重点比较：

- 是否稳定得到 6 个相似视频组。
- 每组成员是否与竞品相同。
- 最大组是否仍包含明显不同的视频。
- 31 个视频是否仍全部进入 Similar Videos。

如果组数为 6 但成员不同，下一步需要保存真实 `date_added`，替换当前
`media_store_id DESC` 近似排序。

不应调整照片/视频 Hash 阈值，因为本次问题已证明来自帧匹配和锚点顺序，而不是阈值。
