# 问题记录与规避清单

本文档用于长期维护 SimilarScanDemo 和相关 SDK 在媒体扫描、数据库查询、异步刷新、权限链路、分页展示等场景中遇到的问题。

这里记录的“当前处理方式”只是当前项目在当时约束下选择的方案，不代表以后遇到类似问题时必须照搬。后续处理同类问题时，需要结合数据量、设备版本、性能目标、业务体验和代码边界重新选择最佳方案。

## 2. 已记录问题

### 2.1 大结果集查询导致 CursorWindow 读取崩溃

- 状态：已处理，持续关注。
- 影响模块：`similar-scan-core` 数据库查询、Demo 首页和详情页数据加载。
- 现象：加载分类或 Other 资源时出现 `Couldn't read row xxxx, col 0 from CursorWindow`。
- 触发条件：一次性从数据库读取大量资源，尤其是 Other 类别或完整分组资产列表。
- 根因：SQLite CursorWindow 承载能力有限，大量行或较多字段一次性进入 Cursor 后，读取到窗口外数据时可能崩溃。
- 当前处理：首页只加载每组少量预览资源，真实数量和大小通过 SQL 聚合获取；详情页通过分页接口按需加载全部资源。
- 可选方案：减少查询字段；分页加载；分组查询；使用聚合 SQL 返回统计值；为大列表引入 Paging；业务上限制单次展示数量。
- 选择依据：如果页面只需要预览，优先使用预览限制和聚合统计；如果页面需要完整列表，优先分页；如果需要复杂滚动状态，可考虑 Paging。
- 验证方式：使用大图库进入首页、Other 分类详情、相似分组详情，多次上下滑动，确认无 CursorWindow 崩溃。
- 后续注意：不要为了方便在首页恢复一次性加载全部 assets；新增查询时确认是否有 `LIMIT/OFFSET` 或聚合统计。
- 相关位置：`ScanDatabase.loadProductCategoryAssets`、`ScanDatabase.loadGroupAssetsPage`、`ScanDatabase.loadPagedAssets`、`MainActivity`、`GroupDetailActivity`。

### 2.2 数据库关闭时机不当导致 already-closed 崩溃

- 状态：已处理，持续关注。
- 影响模块：Activity 生命周期、异步刷新线程、SDK Client 关闭。
- 现象：页面销毁后后台刷新任务仍在访问数据库，出现 `attempt to re-open an already-closed object: SQLiteDatabase`。
- 触发条件：`onDestroy()` 中关闭 `scanClient`，但渲染线程或分页线程已经提交了数据库读取任务。
- 根因：资源关闭和异步任务之间存在竞态，数据库先关闭，后续任务继续查询。
- 当前处理：Activity 销毁时先移除 UI 回调，再把 `scanClient.close()` 投递到同一个执行队列末尾，确保已提交的读取任务先执行完；分页任务提交失败时清理加载状态。
- 可选方案：引用计数管理数据库；生命周期感知的协程作用域；统一 Repository 层持有单例数据库；取消所有任务后再关闭；不在 Activity 层关闭数据库，交给 Application 生命周期管理。
- 选择依据：如果数据库是页面私有资源，可以使用队列内顺序关闭；如果多个页面共享数据库，更适合 Application 或 SDK 单例管理；如果使用协程，可用结构化并发统一取消和关闭。
- 验证方式：扫描中反复进入退出首页和详情页，横向加载分组资源后立即返回，确认无数据库关闭崩溃。
- 后续注意：新增异步查询时必须确认资源关闭顺序；不要在还有后台任务可能访问数据库时直接 close。
- 相关位置：`MainActivity.onDestroy`、`GroupDetailActivity.onDestroy`、`AndroidSimilarScanClient.close`。

### 2.3 首页预览数量和真实分组数量混淆

- 状态：已处理，持续关注。
- 影响模块：产品分类聚合、首页展示、分组排序。
- 现象：首页只取 2 张预览图后，如果继续使用 `assets.size` 判断分组有效性或计算数量，会导致分组数量、分类数量和真实数据不一致。
- 触发条件：接口为了性能只返回每组 N 个预览资源，但展示层仍把返回 assets 当完整组内数据。
- 根因：预览数据和完整数据语义没有拆分。
- 当前处理：`SimilarGroup.totalAssetCount`、`SimilarGroup.totalSizeBytes`、`SimilarGroup.latestAssetTimeMillis` 由数据库对完整分组聚合得到；`assets` 只代表本次接口返回的展示样本。
- 可选方案：拆分 Summary 模型和 Detail 模型；保留同一个模型但明确 total 与 preview 字段语义；首页只使用分类摘要接口；详情页单独加载完整数据。
- 选择依据：如果 SDK 对外 API 要简单，可保留一个模型但加清晰字段；如果业务复杂，建议拆成 `GroupSummary` 和 `GroupDetail`。
- 验证方式：首页分类数量、大小、排序与详情页总量一致；把 `previewAssetLimit` 改成 0、2、5 时统计值不变化。
- 后续注意：不要用 `group.assets.size` 代表真实组内资源数量，除非当前接口明确加载了完整组。
- 相关位置：`SimilarGroup`、`ProductCategory`、`ProductCategoryBuilder`、`ScanDatabase.loadMatchedGroups`。

### 2.4 首页加载完整资源导致卡顿

- 状态：已处理，持续关注。
- 影响模块：Demo 首页数据展示。
- 现象：首页扫描过程中上下滑动或点击分类偶现卡顿，资源多时加载慢。
- 触发条件：首页每次刷新都加载完整分类、完整分组和大量资源缩略信息。
- 根因：首页展示只需要少量预览，但数据层返回了远超 UI 需要的数据量。
- 当前处理：首页调用分类接口时传入业务配置的预览数量，默认每组只展示 2 个资源；真实数量和大小走数据库聚合。
- 可选方案：首页分类摘要接口；节流刷新；DiffUtil 局部刷新；异步缩略图加载；分页预取；缓存最近一次分类结果。
- 选择依据：如果主要瓶颈是数据库读取，优先减少返回数据；如果主要瓶颈是列表重绘，优先 DiffUtil 和节流；如果主要瓶颈是缩略图，优先图片加载缓存。
- 验证方式：扫描中连续刷新首页、上下滑动和点击分类，无明显卡顿。
- 后续注意：首页不要承担详情页职责，首页只展示摘要和少量预览。
- 相关位置：`MainActivity.loadAndRenderCachedGroups`、`SimilarScanClient.loadProductCategories`。

### 2.5 详情页一次性加载全部资源的性能风险

- 状态：已处理，持续关注。
- 影响模块：分组详情页、Other 分类详情页。
- 现象：资源量较大时，进入详情页加载慢，内存压力变大，滑动首屏等待时间长。
- 触发条件：详情页一次性加载某分类下全部资源，或者一次性加载某个大分组的所有资源。
- 根因：详情页没有分页边界，数据库查询和 UI 绑定都集中在一次操作中。
- 当前处理：平铺分类按页加载；相似分组内资源按组分页加载，横向列表接近末尾时再加载下一页。
- 可选方案：RecyclerView Paging；按日期分段加载；按分组懒加载；预取下一页；限制最大首屏数量。
- 选择依据：如果列表是单列或网格，Paging 更自然；如果是多组横向列表，按组分页更容易保持 UI 结构。
- 验证方式：进入 Other、Other Videos、大相似分组详情页，首屏快速展示，继续滑动可加载后续数据。
- 后续注意：新增详情页时必须定义首屏加载量和翻页策略。
- 相关位置：`GroupDetailActivity`、`GroupAdapter`、`ScanDatabase.loadGroupAssetsPage`。

### 2.6 异步扫描和用户删除同时发生的数据一致性

- 状态：已有机制，持续关注。
- 影响模块：扫描任务、删除任务、数据库状态。
- 现象：扫描过程中用户可能在 App 内删除资源，也可能在系统相册删除资源，若处理不当会出现分组引用已删除资源、UI 展示旧数据或扫描任务失败。
- 触发条件：扫描异步执行期间，资源状态被用户或系统修改。
- 根因：扫描结果和用户操作都在修改同一批媒体资源及本地数据库状态。
- 当前处理：SDK 使用资源状态、扫描 token、数据库状态更新和结果重建来降低并发修改风险；Demo 删除后刷新展示数据。
- 可选方案：删除操作排队到扫描任务之后；资源增加 revision；删除时软删除并二次确认；扫描读取前校验当前 state；UI 层对删除中资源禁用操作。
- 选择依据：如果强调实时交互，适合软删除和状态校验；如果强调实现简单，适合扫描结束后统一处理；如果删除成功率要求高，需要系统删除确认和失败恢复。
- 验证方式：扫描中删除资源、删除后进入详情页、扫描完成后重新加载分类，确认无崩溃、无明显重复或失效引用。
- 后续注意：任何用户操作都不能假设扫描已完成；任何扫描结果落库前都要允许资源已经变化。
- 相关位置：`SimilarMediaScanner`、`ScanDatabase`、Demo 删除业务逻辑。

### 2.7 `forceFull = true` 语义容易误解

- 状态：已说明，持续关注。
- 影响模块：扫描入口、SDK 技术文档、测试预期。
- 现象：测试时传入 `forceFull = true`，但并不是所有资源都重新计算指纹。
- 触发条件：已有资源的媒体信息、版本和指纹状态未变化。
- 根因：当前全量扫描代表重新遍历系统媒体库并校验资源状态，不等于强制丢弃所有已完成指纹。
- 当前处理：全量扫描会遍历全部媒体资源；是否重新计算指纹仍由媒体信息变化、算法版本、状态等条件决定，以避免重复耗时计算。
- 可选方案：增加 `forceRecalculateFingerprint`；清空指纹表后重扫；按算法版本触发重算；仅对异常状态资源重算。
- 选择依据：如果测试算法变化，需要强制重算；如果只是校准媒体库完整性，不应重算全部指纹；如果线上升级算法，推荐使用算法版本触发重算。
- 验证方式：观察扫描日志中的 `visited`、`fingerprinted`、`reused`，确认符合当前扫描语义。
- 后续注意：文档和 UI 文案要避免把 full scan 解释成全部重新计算指纹。
- 相关位置：`SimilarMediaScanner.scan`、`ScanStateStore`、SDK 技术文档。

### 2.8 首次安装权限链路导致扫描状态中断

- 状态：已处理，持续关注。
- 影响模块：Demo 权限申请、扫描入口、首页按钮状态。
- 现象：首次安装授予媒体权限后又弹出通知权限，通知权限返回后界面显示 `Rescan`，扫描没有继续，切后台再回来才恢复。
- 触发条件：媒体权限和通知权限连续请求，Activity 回调顺序影响扫描启动状态。
- 根因：权限申请链路没有把“媒体权限已通过但通知权限仍在处理中”的中间态保存下来。
- 当前处理：Demo 使用 pending 状态串联媒体权限和通知权限，通知权限处理完成后继续启动扫描；拒绝通知权限也不阻断媒体扫描。
- 可选方案：权限状态机；单独引导页完成全部授权后再进入首页；通知权限延后到需要前台服务时再请求；使用 Activity Result API 统一编排。
- 选择依据：如果希望首扫体验顺畅，适合状态机串联；如果产品允许延后通知授权，可减少首次权限打断；如果业务依赖前台服务，通知权限需要尽早处理。
- 验证方式：首次安装分别允许/拒绝通知权限，确认媒体扫描都能继续。
- 后续注意：Android 13+ 通知权限不应阻塞媒体扫描主流程，但会影响前台服务通知体验。
- 相关位置：`MainActivity` 权限处理逻辑、`MediaScanService`。

### 2.9 系统媒体库变化和本地缓存一致性

- 状态：已有机制，持续关注。
- 影响模块：MediaStore 枚举、本地资源表、增量扫描。
- 现象：扫描中或扫描后，用户在系统相册新增、删除资源，App 本地数据可能短时间内不是最新。
- 触发条件：系统媒体库在扫描任务执行期间变化，或者用户只授权部分资源。
- 根因：MediaStore 是外部数据源，本地数据库只是扫描时刻的快照。
- 当前处理：完整授权下扫描结束后清理本地未见资源；部分授权下不把未返回资源直接视为删除；下次扫描会重新对齐系统媒体库。
- 可选方案：监听 MediaStore 变化；扫描中周期性检查变更；扫描结束后立即做一次轻量增量；仅在用户手动刷新时对齐。
- 选择依据：如果产品要求强实时，考虑监听和轻量增量；如果更重视稳定和耗电，保留手动或定期扫描；部分授权场景必须避免误删本地记录。
- 验证方式：扫描中新增/删除媒体，扫描完成后再次触发扫描，确认数据库资源数和系统授权范围一致。
- 后续注意：不能在部分授权场景下用“未枚举到”直接判断系统资源已删除。
- 相关位置：`MediaStoreRepository`、`ScanDatabase.removeAssetsNotSeenInScan`、`media_scan_business_constraints.md`。

### 2.10 视频封面和系统相册不一致

- 状态：已处理过多轮，持续关注。
- 影响模块：视频缩略图获取、相似视频展示、压缩模块视频预览。
- 现象：某些视频在 Demo 中展示的封面与系统相册封面不一致，甚至同一相似分组中部分资源封面错位。
- 触发条件：使用不同缩略图来源、RecyclerView 复用、异步加载结果未校验绑定资源。
- 根因：系统相册、MediaMetadataRetriever、ThumbnailUtils 和 MediaStore 缩略图的帧选择策略可能不同；列表复用时如果未校验 item id，异步回调可能设置到错误 View。
- 当前处理：展示层尽量复用系统缩略图方案，并确保异步加载时绑定当前资源；压缩页面的视频封面与相似识别列表使用一致方案。
- 可选方案：统一使用 `ContentResolver.loadThumbnail`；旧版本使用 `MediaStore.Video.Thumbnails`；失败时再用 MediaMetadataRetriever；展示和指纹计算使用不同缩略图来源；为关键页面禁用或减少复用验证问题。
- 选择依据：如果目标是与系统相册观感一致，优先系统缩略图；如果目标是算法稳定，指纹计算可使用固定规则抽帧；如果目标是排查复用问题，可临时使用简单页面验证。
- 验证方式：对比系统相册截图、Demo 相似视频列表、压缩分组列表和详情页封面。
- 后续注意：展示封面一致性和算法指纹稳定性是两个目标，不要简单混为一个实现。
- 相关位置：视频缩略图加载工具、相似视频列表、压缩页面列表。

## 3. 通用检查清单

### 3.1 数据库查询

- 大列表查询必须明确是否需要分页。
- 首页摘要优先使用 SQL 聚合，不要加载完整资源后再在内存统计。
- Cursor 查询字段只取当前页面需要的字段。
- 查询结果需要排序时，优先让数据库返回稳定排序字段。
- 数据库关闭前确认没有已提交但未执行完的查询任务。

### 3.2 生命周期和异步任务

- Activity 销毁时先停止 UI 回调，再处理后台资源关闭。
- 不同线程访问同一个 SDK Client 时，要明确关闭顺序。
- 任务提交可能失败，需要清理 loading 状态。
- 后台任务回调 UI 前要确认页面仍然有效。
- 扫描、刷新、分页、删除不要互相假设对方已经完成。

### 3.3 首页和详情页展示

- 首页只展示摘要和少量预览。
- 详情页才加载完整数据，并且需要分页或按组懒加载。
- 预览数量不能影响真实数量、真实大小和分组有效性判断。
- 所有分类展示排序要有统一规则，当前业务层按媒体时间降序展示。
- 缩略图异步加载必须校验当前绑定的资源。

### 3.4 权限和授权范围

- Android 13+ 图片、视频权限可能分开授予。
- Android 14+ 可能只有用户选择的媒体资源可见。
- 部分授权下不能把未返回资源直接当作已删除。
- 连续拒绝或不再提醒时，Demo 层需要引导用户进入系统设置。
- 通知权限失败不应直接阻断媒体扫描，但会影响前台服务展示。

### 3.5 扫描中断和外部干扰

- 扫描中 App 内删除资源时，扫描结果落库前要检查资源状态。
- 扫描中系统相册删除资源时，下次扫描应能对齐本地缓存。
- 扫描中新增资源时，可以接受本轮不处理，但下轮扫描必须可发现。
- App 被强杀或崩溃后，下次启动应基于数据库状态和 MediaStore 重新判断，而不是依赖内存状态。
- 对于长耗时任务，日志和 UI 指标要能区分 visited、fingerprinted、reused。
